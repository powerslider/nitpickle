# Commit-msg: generate a conventional-commit message

Draft a commit message for the current changes. Output only the message, ready
to copy. Do not stage, commit, push, or run any git write command. The user runs
the commit.

## Source of truth

The commit-message rules live in `preferences.md`. Read it and follow its
commit-message section exactly.

The format below is the default to apply when no `preferences.md` is present.

## Read the changes

- `git diff --cached` for staged changes. If nothing is staged, fall back to
  `git diff` for the working tree and say which set you used.
- `git status --short` for the full scope, including renames and new files.
- Skim the actual diff, not just the file list. The why lives in what the change
  accomplishes, not the filenames.

## Format

Conventional Commits, strictly: `<type>[optional scope][!]: <description>`, an
optional body, then the footer.

- Types: `feat`, `fix`, `build`, `chore`, `ci`, `docs`, `style`, `refactor`,
  `perf`, `test`. Add a scope only when it sharpens the subject.
- Subject in imperative mood, no trailing period, 72 characters or fewer.
- Body explains the **why**, not the what, as bullet points, one per distinct
  reason. Wrap at 72 columns. The diff already shows what changed.
- Breaking changes use `!` after the type or scope, or a `BREAKING CHANGE:`
  footer.
- House style: no em dashes, no semicolons, no AI or tool attribution. No
  `Co-Authored-By`, no "generated by" lines.

## Footer

The footer contains an optional issue reference followed by a sign-off:

```
resolves <issue_id>

Signed-off-by: <first_name> <last_name> (<email>)
```

- `<issue_id>` is the tracked issue for the change. Derive it from the branch
  name or the task context. If none is known, skip the `resolves` line and write
  a footer with the sign-off only.
- The sign-off name and email come from git config (`git config user.name`,
  `git config user.email`). Split the configured name into first and last. Do not
  invent them.

## One change per message

If the diff spans unrelated changes, say so and suggest splitting into separate
commits with a message for each, rather than one message that lists everything.

## Output

Present the message in a fenced block so it is easy to copy. Stop there. The user
runs the commit themselves.
